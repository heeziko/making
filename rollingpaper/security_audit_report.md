# GitHub 배포 보안 점검 결과 보고서 (Security Audit Report)

본 프로젝트의 소스코드가 GitHub 등 공개 저장소(Public Repository)에 업로드되어 배포되기에 앞서, 민감 정보 유출 방지 및 주요 웹 취약점 대응 현황을 종합적으로 점검한 보고서입니다.

---

## 🔍 1. 민감 정보 하드코딩 분석 (Hardcoded Credentials)

### 점검 결과: **안전 (조치 완료)**

- **DB 접속 정보 노출 차단**: 
  - `config/db.php` 파일 내부의 호스트 주소, 계정 ID, 비밀번호 등은 외부에 노출될 경우 DB 탈취로 이어질 수 있는 위험 요소입니다.
  - **조치 현황**: 프로젝트 루트에 `.gitignore` 파일을 생성하여 `config/db.php` 파일이 Git 추적 대상에서 제외되도록 처리하였고, 대신 빈 더미 값으로 채워진 안전한 `config/db.example.php` 설정 템플릿을 생성하여 배포를 준비하였습니다.
- **서버 IP/도메인 유출 점검**:
  - `index.php` 및 `js/app.js` 등 클라이언트 파일 내부에 API를 호출하기 위한 서버 IP(`13.125.143.67`)가 하드코딩되어 있는지 검사하였습니다.
  - **조치 현황**: 모든 API 호출을 특정 IP 주소가 아닌 **상대 경로**(`api/get_messages.php`, `api/create_message.php`) 형식으로 작성하여, 코드 유출 시에도 프로덕션 서버 IP가 깃허브에 노출되지 않도록 완벽히 예방 조치했습니다.
- **SSH Key 노출 점검**:
  - AWS Lightsail 접속용 프라이빗 키 파일(`.pem`)이 실수로 프로젝트 루트 디렉토리 내에 유입되어 있는지 스캔했습니다.
  - **조치 현황**: 로컬 키 파일은 별도 경로(`C:\Users\고희지\Desktop\SSHkey.pem`)에 보관 중이며, 프로젝트 디렉토리 내부에는 키 파일이 전혀 존재하지 않습니다. 또한 `.gitignore`에 `.pem`, `.key` 파일 필터링 규칙을 추가하여 향후 유입 가능성까지 차단하였습니다.

## 🔒 2. 주요 취약점 분석 및 웹 보안 점검

### ① SQL 인젝션 (SQL Injection) 방지
- **점검 결과**: **매우 안전**
- **근거**: 
  - `api/create_message.php` 및 `api/get_messages.php` 내의 모든 SQL 쿼리는 문자열 결합 방식이 아닌, **PDO Prepared Statements (준비된 컴파일 쿼리)**를 사용하여 안전하게 작성되었습니다.
  - `sender`, `content`, `limit`, `offset` 등 사용자 입력 매개변수들은 `$stmt->bindValue`와 `$stmt->execute([...])` 과정을 통해 쿼리 해석기와 변수 영역이 엄격히 분리되므로, 임의의 SQL 코드를 인젝션하는 모든 공격 시도가 무력화됩니다.

### ② 크로스 사이트 스크립팅 (XSS) 방지
- **점검 결과**: **매우 안전 (2중 필터링 적용)**
- **근거**:
  - **서버사이드 필터링**: `api/create_message.php`에서 DB에 데이터를 저장하기 전에 `strip_tags()` 함수를 통과시켜 유해한 HTML 태그 및 `<script>` 태그를 원천적으로 제거 및 여과합니다.
  - **클라이언트사이드 이스케이프**: `js/app.js`에서 카드를 화면에 그리기 전 `escapeHTML` 함수를 구현하여 `&`, `<`, `>`, `"`, `'` 기호를 HTML 특수 엔티티 문자로 완전히 변환한 후 출력합니다. 이를 통해 악성 스크립트가 DOM 상에서 임의로 실행되는 현상을 완벽히 방어합니다.

### ③ 비정상 입력값 및 리소스 거부 (DDoS 대비)
- **점검 결과**: **안전**
- **근거**:
  - `api/create_message.php` 서버 코드에서 `sender`(50자 초과), `content`(1000자 초과)의 길이를 서버사이드에서 유효성 검사하여 400 Bad Request로 차단하므로, 비정상적으로 긴 텍스트 입력으로 인한 메모리 고갈 및 DB 리소스 오버헤드를 사전에 예방합니다.

## ⚠️ 3. 실서비스 배포 시 추가 권한 제한 권장사항 (CORS 정책)

- 현재 백엔드 API인 `get_messages.php` 및 `create_message.php`는 원활한 로컬 개발과 크로스 도메인 테스트를 위해 CORS 헤더가 다음과 같이 모두 허용(`*`)으로 열려 있습니다:
  ```php
  header('Access-Control-Allow-Origin: *');
  ```
- **권장 보안 패치**:
  - 추후 지인 배포 단계를 넘어 상용 상용화 배포 시에는, 악의적인 크로스 오리진 요청(API 난사 및 스패밍 공격)을 예방하기 위해 특정 도메인 주소로 오리진을 제한하는 방향을 권장합니다:
  ```php
  // 상용 배포 시 예시 (실제 도메인이 있을 경우)
  header('Access-Control-Allow-Origin: https://yourdomain.com');
  ```

## 📋 4. 결론 (배포 가능 여부)

- **배포 적합도**: **적합 (Pass)**
- **이유**: 소스코드 내부에 어떠한 중요 크리덴셜(서버 패스워드, API Key, 호스트 IP, SSH Key 등)도 노출되어 있지 않으며, Git 제외 처리(.gitignore) 및 환경 템플릿(db.example.php) 구성이 아주 잘 설계되어 있습니다. 핵심적인 보안 필터링(PDO, XSS Escape)이 모두 내장되어 있으므로 즉시 GitHub Public 저장소에 업로드하셔도 보안상 안전합니다.
